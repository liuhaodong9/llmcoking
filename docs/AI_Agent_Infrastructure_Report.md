# 焦化配煤智能体（CokeAgent）技术架构报告

## 一、概述

本项目构建了一个面向焦化配煤场景的 AI Agent 系统，能够自主完成**煤种查询、配煤方案优化、焦炭质量预测**等任务。系统采用本地大语言模型（Qwen3-8B）驱动 Tool-Calling 决策，结合机器学习预测模型和数学优化算法，形成完整的智能决策闭环。

本文档从 **AI Agent 基础设施的六层架构**出发，阐述系统的设计思路与实现细节。

---

## 二、六层基础设施架构

### 2.1 上下文工程（Context Engineering）

> **核心思想：AI 看不到的信息等于不存在，必须主动把关键知识喂给它。**

| 实现方式 | 说明 |
|---|---|
| 动态 System Prompt | 每轮对话重新构建，注入当前料斗状态、可用煤种数量等实时信息 |
| 数据库驱动 | 煤样属性（106种）从 MySQL `coaldata_table` 实时加载，而非硬编码 |
| 精简上下文 | 工具返回结果只包含必要字段（煤种名、价格、关键指标），避免上下文溢出 |
| `/nothink` 指令 | 对 Qwen3 禁用内部思维链，减少无效 token 消耗，加快响应 |

**System Prompt 示例片段：**
```
你是 DeepCoke 配煤优化智能助手。你可以：
1. 查询煤种 — 列出数据库中所有可用煤种及其化验指标
2. 优化配煤 — 根据焦炭质量约束，找到成本最低的配煤方案
3. 预测质量 — 用机器学习模型预测焦炭的 CRI 和 CSR
```

---

### 2.2 工具设计（Tool Design）

> **核心思想：工具不是越多越好，要精挑细选；操作环境要隔离，防止误操作。**

系统为 LLM 提供 **3 个精选工具**，覆盖配煤全流程：

| 工具 | 功能 | 输入 | 输出 |
|---|---|---|---|
| `list_coals` | 查询可用煤种 | 无 | 煤种列表 + 关键属性 |
| `optimize_blend` | 优化配煤方案 | 煤种名单 + 约束条件 | 配比、重量、成本 |
| `predict_quality` | 预测焦炭质量 | 配煤比例 + 模型选择 | CRI、CSR 预测值 |

**隔离设计：**
- 工具仅执行**只读查询**和**纯计算**，不直接操作物理设备
- 数据库连接只有 SELECT 权限的查询
- 优化算法在内存中运行，不修改任何持久化数据

**工具调用流程：**
```
用户提问 → LLM 分析意图 → 选择工具 + 生成参数(JSON)
    → 执行工具 → 返回结果 → LLM 组织回答
    → 如需更多信息 → 继续调用工具（最多6轮）
```

---

### 2.3 验证机制（Verification）

> **核心思想：不能光靠 AI 说"我做对了"，要有自动化检查。**

| 验证层 | 实现 |
|---|---|
| 约束校验 | 差分进化优化器内置 `NonlinearConstraint`，确保 CRI/CSR/M10/M25 在目标范围内 |
| 配比合法性 | 配比之和强制等于 100%（等式约束），单煤种 0%~60%（边界约束） |
| 无解检测 | 优化器 `result.success == False` 时返回明确错误，不编造结果 |
| 双算法交叉验证 | 差分进化（DE）为主 → 线性规划（LP）为备选，两种算法互相兜底 |
| 模型归一化 | 输入特征经 Min-Max 标准化（硬编码上下界），防止异常值导致预测偏差 |

```python
# 优化器约束示例
NonlinearConstraint(lambda x: np.sum(x), 1.0, 1.0)          # 配比之和 = 1
NonlinearConstraint(lambda x: np.dot(cri, x), CRI_min, CRI_max)  # CRI 范围
```

---

### 2.4 状态管理（State & Progress）

> **核心思想：AI 没有记忆，要用外部机制记录进度，中断后能恢复。**

| 机制 | 实现 |
|---|---|
| 多轮对话历史 | `messages` 列表完整保存每轮 user → assistant → tool 消息链 |
| 工具结果持久化 | 每次工具调用的结果以 `{"role": "tool", "content": ...}` 追加到历史 |
| 最大轮次保护 | `max_turns=6` 防止无限循环，超限后强制要求 LLM 总结 |
| 数据缓存 | 煤样属性首次查询后缓存在内存，避免重复数据库访问 |
| Pipeline 进度流 | 前端实时展示推理链路（问题分类 → Agent 调用 → 完成） |

**对话状态流转：**
```
Turn 1: User问"优化配煤" → LLM调list_coals → 获取煤种列表
Turn 2: LLM调optimize_blend(coal_names, constraints) → 获取最优方案
Turn 3: LLM调predict_quality(blend_ratios) → 验证预测质量
Turn 4: LLM综合所有结果 → 输出最终方案表格
```

---

### 2.5 可观测性（Observability）

> **核心思想：记录每一步做了什么，出错能追溯，下次能改进。**

| 层级 | 实现 |
|---|---|
| Pipeline 日志 | 每个步骤打印 `[PIPELINE] Step N: ...`，含耗时和结果摘要 |
| Agent 日志 | 工具调用记录 `Tool call: optimize_blend({...})` |
| 前端进度展示 | Markdown 折叠块实时展示推理链路，用户可看到每步状态 |
| 错误分级 | 致命错误（LLM连接失败）终止流程，非致命错误（KG连接异常）跳过并标记 |

**前端进度展示效果：**
```markdown
<details open>
<summary>推理链路</summary>

> ✅ **问题类型：** 配煤优化
> ⚙️ **正在调用配煤优化 Agent…**
> ✅ **配煤优化 Agent 完成**
</details>
```

---

### 2.6 人机协作（Human-in-the-Loop）

> **核心思想：关键决策让人拍板，AI 不能越权。**

| 设计 | 说明 |
|---|---|
| Agent 只做建议 | 输出配煤方案和预测结果，**不直接下发到 PLC/MQTT** |
| 执行需确认 | 独立的 CokeAgent 版本中 `execute_blend` 工具需要人工触发 |
| MQTT 可选 | MQTT 客户端 import 失败时优雅降级，不影响计算功能 |
| 约束由人设定 | CRI/CSR/M10/M25 的目标范围由操作员输入，AI 不自行决定 |

## 五、核心模块说明

### 5.1 质量预测模型

输入 7 维煤质特征（加权平均后 Min-Max 归一化），预测焦炭反应性（CRI）和反应后强度（CSR）。

| 模型 | 文件 | 输出 |
|---|---|---|
| 随机森林 (RF) | RF.pickle | CRI, CSR |
| K近邻 (KNN) | KNN.pickle | CRI, CSR |
| 线性回归 | linear.pickle | CRI, CSR |
| 决策树 | decisiontree.pickle | CRI, CSR |
| 支持向量回归 | SVR_CRI/CSR.pickle | CRI, CSR (分别建模) |
| 梯度提升 (GBR) | GBR_1/2.pickle | CRI, CSR (分别建模) |

### 5.2 配煤优化算法

**目标函数：** 最小化配煤成本（价格加权和）

**约束条件：**
- 等式约束：配比之和 = 100%
- 边界约束：单煤种 0% ~ 60%
- 非线性约束：CRI/CSR/M10/M25 加权平均值在目标范围内
- 线性约束（LP）：Vdaf ≤ 上限，G ≥ 下限，Ad ≤ 上限

**算法流程：**
1. 有质量约束 → 差分进化（DE），popsize=100，maxiter=2000
2. DE 无解 → 退回线性规划（LP）
3. LP 也无解 → 返回错误提示

## 六、焦炭质量行业标准

### 6.1 国标分级（GB/T 4000）

| 等级 | CRI (%) | CSR (%) | M25 (%) | M10 (%) | 灰分 Ad (%) | 硫分 S (%) |
|---|---|---|---|---|---|---|
| 一级冶金焦 | ≤28 | ≥64 | ≥92.0 | ≤7.0 | ≤12.0 | ≤0.6 |
| 二级冶金焦 | ≤30 | ≥58 | ≥88.0 | ≤8.0 | ≤13.5 | ≤0.8 |
| 三级冶金焦 | ≤33 | ≥55 | ≥83.0 | ≤10.0 | ≤15.0 | ≤1.0 |

> **参考标准：** ISO 18894:2018、GB/T 4000-2017

### 6.2 指标含义

| 指标 | 全称 | 含义 | 优劣方向 |
|---|---|---|---|
| CRI | 焦炭反应性 | 焦炭与 CO₂ 在 1100°C 反应 2h 的质量损失率 | **越低越好** |
| CSR | 反应后强度 | 反应后焦炭在转鼓试验中的强度 | **越高越好** |
| M25 | 机械强度(>25mm) | 转鼓后 >25mm 粒级百分比 | **越高越好** |
| M10 | 耐磨强度(<10mm) | 转鼓后 <10mm 粉末百分比 | **越低越好** |

> CRI 与 CSR 呈显著负相关：CRI 越低，CSR 通常越高。

### 6.3 本系统煤样实际水平

本系统数据库中 15 种煤样的焦炭指标范围：
- **CRI：** 20 ~ 54（均值 ≈ 38）
- **CSR：** 22 ~ 60（均值 ≈ 39）

大部分煤样焦炭指标介于二级~三级水平，仅"美国1"(CRI=20, CSR=60) 接近一级标准。因此演示时约束应设置在合理范围内（如 CRI<35, CSR>40）。

## 七、演示场景

### 场景1：查询可用煤种
```
用户：帮我看看有哪些煤可以用

Agent：调用 list_coals → 返回15种煤的表格（名称、类型、价格、Vdaf、G、Ad、CRI、CSR）
```

### 场景2：优化配煤方案
```
用户：帮我用晋茂、美国1、赵城、盛泰源、红果优化配煤，要求 CRI 小于 35，CSR 大于 40

Agent：
  1. 调用 optimize_blend(coal_names=["晋茂","美国1","赵城","盛泰源","红果"], CRI_max=35, CSR_min=40)
  2. 返回最优配比方案（赵城60%为主）+ 成本 759元/吨
```

### 场景3：预测焦炭质量
```
用户：如果晋茂30%、美国1 20%、赵城30%、盛泰源20%，预测一下焦炭质量

Agent：
  1. 调用 predict_quality(blend_ratios={晋茂:30, 美国1:20, 赵城:30, 盛泰源:20}, model_name="RF")
  2. 返回预测结果：CRI=37.14, CSR=44.58, Ad=12.1
```

### 场景4：多模型对比预测
```
用户：用所有模型分别预测一下这个配比的焦炭质量，做一个对比

Agent：依次用 RF/SVR/KNN/GBR/Linear/DecisionTree 6个模型预测，返回对比表格：
  - CRI 范围：31.4 ~ 37.14
  - CSR 范围：43.36 ~ 46.54
  - DecisionTree 预测 CRI 最低(31.4)，GBR 预测 CSR 最高(46.54)
```

### 场景5：RAG 文献知识问答
```
用户：焦化过程中温度对焦炭质量有什么影响？

RAG Pipeline：
  1. 分类为"因果推理" → 走 RAG + ESCARGOT 深度推理
  2. 检索到 10 条相关文献片段（ChromaDB 6017条向量库）
  3. 生成带文献引用 [1][2][3] 的专业回答
  4. 自动生成 3 个延伸问题
```

## 八、多 Agent 架构设计

### 8.1 为什么采用多 Agent 而非单 Agent

| 对比项 | 单 Agent + 多 Tool | 多 Agent（当前方案） |
|---|---|---|
| LLM 实例 | 1个，共享上下文 | 各自独立，互不干扰 |
| System Prompt | 臃肿（配煤+RAG规则全塞一起） | 各自精简，专注本职 |
| 上下文窗口 | 容易溢出（4096 token 不够） | 各Agent独立使用，互不抢占 |
| 错误隔离 | 一个模块崩溃影响全局 | RAG 挂了不影响配煤，反之亦然 |
| 可维护性 | 所有逻辑耦合在一起 | 模块独立，可单独升级 |

> 8B 小模型注意力有限，上下文塞太多不相关信息会显著降低每个任务的质量。多 Agent 让每个 Agent 只看自己需要的信息。

### 8.2 当前架构

```
用户提问
  │
  ▼
┌─────────────┐
│  Router      │  关键词匹配（0延迟，不依赖LLM）
│  (分类器)     │
└──┬──────┬───┘
   │      │
   ▼      ▼
┌──────┐ ┌──────────┐
│Coal  │ │RAG Agent │
│Agent │ │          │
│      │ │ ChromaDB │
│Tools:│ │ Neo4j    │
│-list │ │ ESCARGOT │
│-opt  │ │ 文献生成  │
│-pred │ │          │
└──────┘ └──────────┘
```

- **Coal Agent**：专注配煤操作（查询、优化、预测），用 Ollama 原生 API + Tool-Calling
- **RAG Agent**：专注知识问答（文献检索、推理、引用），用流式生成 + 文献引用
- **Router**：基于关键词匹配的分流器，零延迟，不消耗 LLM 算力
